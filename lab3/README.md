# Lab 3: GitHub Actions CI/CD (Замена GitLab)

Так как регистрация в GitLab требует привязки иностранной карты, эта лабораторная работа была полностью переделана под **GitHub Actions**. 

## Подробная инструкция по запуску:

### Шаг 1: Подготовка репозитория на GitHub
1. Перейдите на сайт [GitHub](https://github.com/) и авторизуйтесь.
2. Создайте новый пустой репозиторий (кнопка **New** на главной странице). Назовите его, например, `devops_airflow_spark`.
3. Откройте терминал на вашем компьютере и склонируйте этот пустой репозиторий:
   ```bash
   git clone https://github.com/ВАШ_ЛОГИН/devops_airflow_spark.git
   cd devops_airflow_spark
   ```

### Шаг 2: Перенос файлов
Вам нужно собрать в этом репозитории файлы из предыдущих лабораторных работ (по сути, код вашего проекта). 
1. Скопируйте в эту папку (`devops_airflow_spark`) следующие файлы и папки из папки `lab2`:
   - Папку `dags` (с вашими дагами)
   - Папку `spark` (со скриптом my_script.py)
   - Файл `Dockerfile`
   - Файл `docker-compose.yml`
2. Из папки `lab3` скопируйте папку `.github` целиком в корень вашего нового репозитория. Должна получиться структура `.github/workflows/main.yml`.

### Шаг 3: Отправка изменений (Push)
1. Находясь в папке репозитория `devops_airflow_spark`, выполните команды для отправки кода на GitHub:
   ```bash
   git add .
   git commit -m "Initial commit with Airflow, Spark and GitHub Actions pipeline"
   git push origin main
   ```

## Что проверить и как показать результат:
1. Зайдите в ваш репозиторий на сайте GitHub.
2. Перейдите на вкладку **Actions** в верхней панели.
3. Вы увидите запущенный пайплайн с названием вашего коммита. Кликните по нему.
4. Внутри вы увидите визуализацию из трех шагов: `test`, `build` и `deploy`.
5. **Проверка правил пайплайна:**
   - **main ветка:** По умолчанию вы запушили в ветку `main`, поэтому должны успешно отработать все три шага: `test`, `build`, `deploy`.
   - **feature ветка:** Чтобы проверить работу условий, создайте новую ветку, начинающуюся на `feature/`.
     В терминале:
     ```bash
     git checkout -b feature/test-branch
     # Сделайте любое мелкое изменение, например, добавьте пустую строку в любой файл
     git add .
     git commit -m "Test feature branch pipeline"
     git push origin feature/test-branch
     ```
     Снова зайдите во вкладку **Actions**. Вы увидите новый запуск пайплайна, но в этот раз шаг `build` будет пропущен (помечен серым значком skipped), потому что мы настроили такое правило для веток `feature/*`.

Это полностью покроет все требования из оригинальной методички по 3-й лабораторной работе (наличие стейджей тестов, билда, деплоя и условий для веток).
